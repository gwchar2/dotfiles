# Nix, macOS, and NixOS Migration Plan

## Goal

Make this dotfiles repo a clear, reproducible setup for:

- a brand new macOS machine as the primary daily driver
- NixOS as a first-class Linux target
- WSL/Windows as a supported fallback path
- shared terminal, editor, shell, AI, and agent workflow configuration
- a small, boring, easy-to-edit layout that does not preserve complexity just
  because it already exists

The target outcome is one understandable repo where each layer has a clear owner:
system settings, packages, user config, application config, AI instructions, and
bootstrap/rebuild commands.

This migration is also a simplification project. Do not just wrap the current
repo in Nix. Prefer Kun's clean shape where user files live under `home/`, app
configs live where the apps expect them, and Nix declares how those files are
installed.

## Current State

The repo is currently script-first:

- `bootstrap.sh` dispatches into `scripts/install.sh`.
- `scripts/macos.sh` assumes Homebrew already exists, then runs
  `brew bundle --file homebrew/Brewfile`.
- `scripts/wsl.sh` installs Ubuntu/WSL packages through `apt` plus direct curl
  installers for tools like Node, Neovim, Rust, .NET, Starship, and CodeRabbit.
- `scripts/link.sh` symlinks config directories into `$HOME`.
- `scripts/ai.sh` installs optional AI CLIs and copies/symlinks shared agent
  instructions, skills, and rules.

This works, but it is not fully declarative. It installs a lot of things, but it
does not describe the entire desired state of a macOS or NixOS machine.

The repo is also too broad for the desired daily workflow. Several root-level
config folders exist because tools were added incrementally:

- `starship/`
- `yazi/`
- `tmux/`
- `wezterm/`
- `nvim/`
- `zsh/`

Some of these may still be useful, but their current placement makes the repo
feel larger than the actual workflow. During migration, every tool should earn
its place. If a tool is not part of the desired daily loop, remove or archive its
config instead of carrying it forward.

## Tools To Use

### Nix

Nix is the package manager and build system. It provides reproducible package
sets and lockable inputs.

What it requires:

- Nix installed on macOS or NixOS.
- flakes enabled. Determinate Nix handles this cleanly on macOS.
- a `flake.nix` file at the repo root.
- a committed `flake.lock` once inputs are resolved.

Where files should sit:

- `flake.nix`: root entrypoint for all Nix outputs.
- `flake.lock`: root lockfile generated by Nix.
- `nix/`: repo-local Nix modules and shared package lists.

What it owns:

- versions of nixpkgs, nix-darwin, home-manager, nix-homebrew, and other Nix
  inputs
- top-level machine outputs such as macOS, NixOS, and standalone home-manager
  configs

### nix-darwin

nix-darwin applies macOS system configuration with Nix.

What it requires:

- macOS.
- Nix installed.
- administrator permission for `darwin-rebuild switch`.

Where files should sit:

- `nix/darwin/default.nix`: macOS system module.
- optional host-specific files under `hosts/<hostname>/darwin.nix`.

What it owns:

- macOS defaults such as dark mode, key repeat, Dock autohide, Finder behavior,
  menu bar autohide, tap-to-click, and desktop icon visibility
- system-level macOS users and platform settings
- nix-homebrew module wiring

What it should not own:

- application config contents like Neovim Lua or WezTerm Lua
- project templates
- secrets

### home-manager

home-manager manages user-level packages, shell configuration, and dotfile links.
It should be the main bridge between macOS and NixOS.

What it requires:

- Nix.
- either integration through nix-darwin/NixOS, or standalone activation.

Where files should sit:

- `nix/home/default.nix`: shared user configuration.
- `nix/home/packages.nix`: shared user package list.
- `nix/home/shell.nix`: zsh, aliases, Starship, fzf, zoxide.
- `nix/home/files.nix`: symlinks into this repo.
- `nix/home/ai.nix`: agent instruction files and tool-specific paths.

What it owns:

- zsh enablement and shell plugins
- Starship configuration ownership
- user packages like `ripgrep`, `fd`, `fzf`, `jq`, `lazygit`, `neovim`, `yazi`
- symlinks for config directories
- shared AI instruction file placement

What it should not own:

- macOS Dock/Finder settings
- NixOS hardware/system boot config
- Windows PowerShell setup

### nix-homebrew

nix-homebrew installs and wires Homebrew declaratively on macOS.

What it requires:

- macOS.
- nix-darwin integration.
- a Homebrew package/cask list declared in Nix.

Where files should sit:

- `nix/darwin/homebrew.nix` or inside `nix/darwin/default.nix`.

What it owns:

- Homebrew installation
- Homebrew taps if needed
- Homebrew brews and casks
- cleanup policy

Important decision:

- `cleanup = "zap"` is highly reproducible but destructive to undeclared
  packages. Start conservative, then move to strict cleanup after the package
  list is complete.

### NixOS Modules

NixOS modules configure actual NixOS machines.

What they require:

- NixOS installed or accessible.
- host hardware config generated by NixOS.
- root/admin access for `nixos-rebuild switch`.

Where files should sit:

- `hosts/<hostname>/configuration.nix`: host system config.
- `hosts/<hostname>/hardware-configuration.nix`: generated host hardware file.
- `nix/nixos/default.nix`: shared NixOS module imported by hosts.

What they own:

- users
- shell defaults
- system packages
- fonts
- services
- display/session settings if relevant
- NixOS-specific clipboard and terminal dependencies

What they should not own:

- personal editor config internals
- macOS defaults

### nh

`nh` is a nicer command wrapper for NixOS/home-manager rebuilds.

What it requires:

- Nix.
- the `nh` package installed.

What it owns:

- command ergonomics only

Use it as convenience, not as the architecture. The underlying `darwin-rebuild`,
`nixos-rebuild`, and `home-manager` commands should remain understandable.

### nix-direnv

`nix-direnv` loads project-specific Nix dev shells automatically.

What it requires:

- `direnv`.
- `nix-direnv`.
- shell hook enabled.
- each project can opt in with `.envrc`.

What it owns:

- per-project development environments

Use it later after the machine-level setup is stable.

### sops-nix or agenix

These manage secrets declaratively.

What they require:

- a decision about key material and where secrets are allowed to live.
- encrypted secret files committed to the repo.

What they own:

- secrets such as API tokens, if we decide to manage them declaratively.

Do not start here. Add this after the base system is reproducible.

### mas

`mas` installs Mac App Store applications.

What it requires:

- signed-in App Store account.
- app numeric IDs.

What it owns:

- App Store app installation.

Use only if we need App Store apps in the bootstrap.

### herdr

`herdr` is the intended default terminal multiplexer for the new setup. tmux can
remain temporarily as a fallback during migration, but the goal is not to keep
two equal multiplexing systems forever.

What it requires:

- install via Homebrew on macOS, or available package path on other systems.
- config under `~/.config/herdr/config.toml`.

What it owns:

- terminal multiplexing workflow
- agent session visibility if its integrations are better than tmux

Open decision:

- what exact herdr keybindings preserve the useful parts of the current tmux
  muscle memory
- whether herdr is available and stable enough on NixOS for tomorrow's setup
- how Windows/WSL should behave if herdr is not available there

## Target File Layout

Proposed structure:

```text
.
├── flake.nix
├── flake.lock
├── rebuild.sh
├── bootstrap.sh
├── home/
│   ├── AGENTS.md
│   ├── .claude/
│   │   └── settings.json
│   └── .config/
│       ├── nvim/
│       ├── wezterm/
│       ├── herdr/
│       ├── zsh/
│       └── starship/        # only if we keep Starship
├── nix/
│   ├── darwin/
│   │   ├── default.nix
│   │   ├── homebrew.nix
│   │   └── defaults.nix
│   ├── home/
│   │   ├── default.nix
│   │   ├── packages.nix
│   │   ├── shell.nix
│   │   ├── files.nix
│   │   └── ai.nix
│   ├── nixos/
│   │   └── default.nix
│   └── shared/
│       ├── packages.nix
│       └── users.nix
├── hosts/
│   ├── mac/
│   │   └── default.nix
│   └── nixos/
│       ├── default.nix
│       └── hardware-configuration.nix
├── .agents/
└── scripts/
```

This moves real application configs under `home/`, mirroring their installed
locations. It should feel obvious that:

- `home/.config/nvim` becomes `~/.config/nvim`
- `home/.config/wezterm` becomes `~/.config/wezterm`
- `home/.config/herdr` becomes `~/.config/herdr`
- `home/.claude/settings.json` becomes `~/.claude/settings.json`
- `home/AGENTS.md` fans out to supported agent instruction paths

The root of the repo should mostly contain orchestration and documentation, not
one folder per app.

## Tool Retention Review

Before moving each current config, decide whether the tool belongs in the new
daily workflow.

### Keep By Default

- Neovim: primary editor.
- WezTerm: terminal emulator on macOS and Windows.
- herdr: target multiplexer/session manager.
- zsh: shell.
- shared agent instructions and skills.
- Homebrew/Nix package declarations.

### Question Before Keeping

- Starship: useful if prompt customization matters, but it is another config
  surface. If the desired prompt is simple, use a small home-manager Starship
  declaration or remove it.
- Yazi: useful if it is actually part of daily navigation. If Neovim Oil and
  Telescope cover file work, drop Yazi from the default setup.
- tmux: keep only as a migration fallback until herdr is proven.
- large Neovim plugin set: keep core editing features, but review UI and workflow
  plugins that are not part of the actual daily loop.

### Remove Or Archive By Default

- configs for tools that are no longer in the default workflow
- duplicate package installation paths once Nix owns macOS/NixOS
- scripts that only exist to compensate for missing Nix ownership

## Configuration Ownership

### `flake.nix`

Declare:

- inputs: `nixpkgs`, `nix-darwin`, `home-manager`, `nix-homebrew`
- outputs:
  - `darwinConfigurations.mac`
  - `nixosConfigurations.<host>` when the NixOS host is known
  - optional `homeConfigurations.<user>@nixos`
- shared variables:
  - username
  - repo path convention
  - system architecture

Avoid:

- long package lists
- tool-specific config

### `nix/darwin/default.nix`

Declare:

- macOS platform
- primary user
- system state version
- imports for macOS defaults and Homebrew

### `nix/darwin/defaults.nix`

Declare:

- dark mode
- fast key repeat
- short initial repeat delay
- Dock autohide
- menu bar autohide
- Finder list view
- show file extensions
- hide desktop icons
- tap-to-click

### `nix/darwin/homebrew.nix`

Declare:

- Homebrew enabled through nix-homebrew
- brews translated from `homebrew/Brewfile`
- casks translated from `homebrew/Brewfile`
- cleanup policy

### `nix/home/default.nix`

Declare:

- imports for shared user modules
- username and home directory
- home-manager state version

### `nix/home/packages.nix`

Declare:

- shared CLI packages available on macOS and NixOS
- fonts where Nix can own them
- only packages that are part of the default workflow

Examples:

- `ripgrep`
- `fd`
- `fzf`
- `jq`
- `lazygit`
- `neovim`
- `eza`
- `bat`
- `zoxide`
- `herdr`, if available through the selected package channel

### `nix/home/shell.nix`

Declare:

- zsh enablement
- autosuggestions
- syntax highlighting
- only aliases that are used daily
- Starship enablement only if we keep Starship
- fzf and zoxide shell integration
- environment variables like `EDITOR=nvim`

Decision:

- either keep `zsh/*.zsh` as source files and link them, or express most shell
  behavior in home-manager. Prefer incremental migration.
- avoid a sprawling shell layer. If an alias or helper is rarely used, do not
  migrate it by default.

### `nix/home/files.nix`

Declare symlinks for:

- `~/.config/nvim` -> repo `home/.config/nvim`
- `~/.config/wezterm` -> repo `home/.config/wezterm`
- `~/.config/herdr` -> repo `home/.config/herdr`
- `~/.config/zsh` -> repo `home/.config/zsh`, if zsh remains file-based
- `~/.config/starship` -> repo `home/.config/starship`, only if Starship stays
- `~/.gitconfig` -> repo `home/.gitconfig`, if we manage Git config as a file

Prefer `mkOutOfStoreSymlink` for edit-in-place config.

### `nix/home/ai.nix`

Declare:

- `~/AGENTS.md`
- `~/.codex/AGENTS.md`
- `~/.claude/CLAUDE.md`
- `~/.cursor/cursor.md`
- `~/.gemini/GEMINI.md`
- supported skills/rules paths if home-manager should own them

Decision:

- symlink these files instead of copying them, unless a tool cannot tolerate
  symlinks.

### `nix/nixos/default.nix`

Declare:

- shared NixOS user packages
- zsh as available shell
- fonts
- clipboard dependencies
- terminal dependencies
- optional services needed for the desktop environment

Host-specific hardware stays under `hosts/<hostname>/hardware-configuration.nix`.

### `scripts/`

Keep for:

- WSL fallback
- Windows WezTerm shim
- legacy install path
- validation scripts
- migration helpers

Do not make scripts the source of truth for macOS/NixOS once Nix is in place.

## Migration Phases

### Phase 0 - Decide Initial Scope

Decisions needed:

- macOS username and host label.
- NixOS target type:
  - full NixOS system config, if we control the machine
  - standalone home-manager, if we only control the user
- Apple Silicon vs Intel Mac.
- Homebrew cleanup policy.
- herdr as intended default, with tmux fallback only if herdr cannot cover a
  required workflow.
- which current root-level app configs should survive the simplification.
- whether Starship and Yazi are part of the new default workflow.

Exit criteria:

- decisions recorded in this file or an ADR.

### Phase 1 - Add Nix Skeleton

Tasks:

- add `flake.nix`
- add `nix/` module directories
- add `darwinConfigurations.mac`
- add shared home-manager module
- add `rebuild.sh`
- keep current scripts untouched
- add `home/` target layout
- start new config ownership under `home/` rather than adding more root-level
  app folders

Validation:

- `nix flake check --no-build`
- dry-run build for Darwin if run on macOS

### Phase 2 - macOS Declarative Setup

Tasks:

- add nix-darwin macOS defaults
- add nix-homebrew
- translate `homebrew/Brewfile` into `nix/darwin/homebrew.nix`
- set Homebrew cleanup to conservative mode first
- declare WezTerm, Claude Code, fonts, and core CLI tools

Validation:

- fresh-machine bootstrap path documented
- `darwin-rebuild switch --flake .#mac` on macOS
- verify Dock/menu bar/Finder/key repeat/tap-to-click

### Phase 3 - Shared User Config

Tasks:

- move user package ownership into home-manager
- move kept app configs under `home/`
- link app configs from `home/` through home-manager
- migrate zsh/Starship integration carefully
- keep Neovim plugin bootstrap unchanged initially
- link AI instruction files declaratively
- remove or stop linking configs for tools we decide not to keep

Validation:

- `zsh -ic 'echo zsh ok'`
- `nvim --headless +qa`
- `wezterm --version`
- `herdr --version`, if available
- `starship --version`, only if Starship stays
- agent instruction paths exist and point to expected source

### Phase 4 - Herdr As Default

Tasks:

- add herdr config under `home/.config/herdr`
- install herdr on macOS through Homebrew declaration
- install herdr on NixOS if available, or document fallback if not
- adjust WezTerm startup to launch herdr instead of tmux on macOS
- migrate only the tmux keybindings/workflows that are still useful
- remove tmux from the default flow after herdr is validated

Validation:

- create tabs/workspaces
- split panes
- run Codex/Claude sessions
- verify copy mode
- verify agent visibility/status behavior
- verify startup from WezTerm

### Phase 5 - NixOS Readiness

Tasks:

- add NixOS host module or standalone home-manager output
- map WSL package list to Nix packages
- configure user shell and fonts
- link same app configs
- document exact NixOS activation command

Validation:

- `nixos-rebuild build --flake .#<host>` for full NixOS
- or `home-manager build --flake .#<user>@nixos` for user-only

### Phase 6 - Cleanup And Strict Reproducibility

Tasks:

- decide whether to enable Homebrew `zap` cleanup
- remove duplicated package ownership
- document migration from scripts to Nix
- keep WSL script path documented as fallback

Validation:

- fresh macOS setup from clone
- NixOS setup from clone
- README reflects actual workflow
- `./scripts/check.sh`

## Immediate Questions Before Implementation

1. What is the macOS username and desired host label?
2. Is tomorrow's NixOS machine fully controlled by us, or do we only get a user
   account?
3. Should Homebrew cleanup start strict or conservative?
4. Should WezTerm stop auto-maximizing immediately, or wait for the Nix migration?
5. Which of these tools are truly default workflow tools: Starship, Yazi, tmux?
6. Should tmux be archived immediately after herdr works, or kept as a documented
   fallback for WSL only?

## Suggested Defaults

- host label: `mac`
- NixOS output label: `nixos`
- Homebrew cleanup: conservative first, strict later
- terminal multiplexer: herdr default, tmux temporary fallback only
- WezTerm: remove startup maximize and set explicit initial rows/columns
- repo layout: move user-facing files under `home/`, mirroring their installed
  paths
- Starship: keep only if the prompt configuration is intentionally useful; if
  kept, prefer a small home-manager declaration over a large standalone folder
- Yazi: remove from the default setup unless it is actively used
- migration style: add Nix alongside current scripts, simplify the layout, then
  retire duplicated script ownership gradually

## Definition Of Done

The migration is complete when:

- a fresh macOS machine can clone the repo, run one bootstrap command, and get
  Nix, Homebrew, macOS defaults, user packages, terminal/editor/shell config,
  and agent instructions
- a NixOS machine can apply either full system config or standalone
  home-manager config from the same repo
- the README documents the new workflows clearly
- current WSL/Windows support remains available
- package ownership is clear and not duplicated accidentally
- validation commands are documented and passing
